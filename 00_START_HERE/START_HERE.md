# Purpose

This document is the entry point to the Nayara Brand OS. It explains how to use the repository, how to navigate the folders, and which documents should be consulted before creating, editing, reviewing, or approving any Brand OS material.

This document does not define Nayara's brand strategy. It defines how the repository should be used.

# How this document connects to the rest of the repository

This document comes first in the repository reading order. It points every user and AI model to the correct navigation path before any work begins.

The correct starting order is:

1. START_HERE.md
2. NON_NEGOTIABLE_RULES.md
3. MASTER_INDEX.md
4. The relevant folder and document for the task

If another document appears to conflict with START_HERE.md, follow START_HERE.md first, then check NON_NEGOTIABLE_RULES.md.

# Core Principles

• The repository stores approved knowledge only.
• Knowledge comes before prompts.
• Strategy comes before execution.
• Nayara's personal brand comes before employer context.
• Pluralsight can support the brand with context, examples, and proof, but it must not define the brand.
• Every Markdown file must follow the same template.
• Every folder is the atomic unit of approval.
• No folder is complete until all documents inside it pass the repository checklist.

# Decision Framework

Before using the repository, decide:

1. What is the task?
2. Which folder contains the relevant knowledge?
3. Which documents are dependencies?
4. Is the task strategic, creative, review based, or archival?
5. Is there enough approved knowledge to proceed?
6. If evidence is missing, should the output be labelled as an observation, opinion, hypothesis, estimate, or prediction?
7. Does the task require a repository update or just a working response?

# Detailed Rules

Repository folders:

• 00_START_HERE contains repository operating rules, navigation, indexing, and change tracking.
• 01_FOUNDATION contains the core personal brand strategy.
• 02_CREATION_SYSTEM contains creation rules for content, hooks, stories, CTAs, and formats.
• 03_REVIEW_SYSTEM contains review, scoring, hallucination prevention, and approval rules.
• 04_EXAMPLES contains approved examples and pattern libraries.
• 05_OUTPUTS contains templates, approved assets, and reusable outputs.
• 99_ARCHIVE contains old versions and historical notes only.

Use the highest level strategic document first.

When writing content, consult the foundation before execution documents.

When editing repository files, consult NON_NEGOTIABLE_RULES.md before changing anything.

When reviewing or approving, consult the review system once it exists.

Edge cases:

If a required folder does not exist yet, identify the gap and do not invent missing standards.

If archived material conflicts with approved material, approved material wins.

If a task asks for speed, still apply the relevant standards in compressed form.

If the repository is being expanded, build one folder at a time.

# Good Examples

Good example:

Task: Create a LinkedIn post about AI readiness.

Correct process:

1. Read START_HERE.md.
2. Read NON_NEGOTIABLE_RULES.md.
3. Read MASTER_INDEX.md.
4. Consult Brand_Positioning.md, Audience_Psychology.md, Voice_Communication_System.md, Knowledge_Boundaries.md, and relevant Creation System files.

Why this works:

It protects brand strategy before execution.

Good example:

Task: Add a new Markdown file.

Correct process:

Use the metadata block, the mandatory twelve section template, the repository checklist, and the workflow before committing the file.

Why this works:

It prevents repository drift.

# Bad Examples

Bad example:

Reading only Hook_Library.md before writing a post.

Why this fails:

A hook can be technically strong but strategically wrong.

Bad example:

Using a Pluralsight proof point as the centre of Nayara's positioning.

Why this fails:

Pluralsight is supporting context, not the personal brand.

Bad example:

Creating a new document with extra sections because it feels useful.

Why this fails:

It breaks the repository standard.

# Common Mistakes

• Starting with prompts instead of knowledge.
• Treating Phase numbers as the final repository structure.
• Uploading random files to Claude without a navigation map.
• Using current role context as the brand identity.
• Skipping dependencies.
• Treating archived material as current guidance.
• Creating a document before the folder is ready to be approved.

# Anti Patterns

• Prompt first, strategy later.
• Product first, personal brand later.
• Drafts saved as repository files.
• One off documents with inconsistent structure.
• Undocumented changes.
• Claims without evidence.
• Examples presented as real when they are invented.
• Moving to the next folder before the current folder is approved.
