# Purpose

This document defines how approved frameworks should be stored inside the Nayara Brand OS. Frameworks are reusable thinking tools that help Nayara explain decisions, risks, capability gaps, and business consequences.

# How this document connects to the rest of the repository

Framework_Index.md belongs inside 05_OUTPUTS because it stores reusable assets derived from the Brand OS. Frameworks should come from the Foundation and Creation System, then pass the Review System before being stored here.

# Core Principles

• A framework should clarify a decision.
• A framework should be simple enough to remember.
• A framework should support Nayara's positioning.
• A framework should be useful across multiple content formats.
• A framework should not pretend to be research unless evidence supports it.

# Decision Framework

Before adding a framework, ask:

1. What decision does this framework help clarify?
2. What audience will use it?
3. Is it distinct from existing frameworks?
4. Does it fit Nayara's brand philosophy?
5. Can it be used in posts, carousels, talks, or advisory work?
6. Has it passed review and approval?

# Detailed Rules

Approved framework categories:

• Capability risk frameworks.
• AI readiness frameworks.
• Workforce decision frameworks.
• Skills gap diagnostic frameworks.
• Learning impact frameworks.
• Buyer psychology frameworks.
• Content creation frameworks.

Rules:

Do not store half formed frameworks here.

Do not store frameworks copied from other creators unless properly credited and used only as reference.

Do not store frameworks that are too abstract to apply.

Each framework should include purpose, use case, steps, example application, and quality checks.

Edge cases:

If a framework is only useful for one post, store it as a post idea instead of a framework.

If a framework is visual, it may also belong in Carousels after approval.

If a framework is borrowed, it must be clearly labelled and not presented as Nayara's original system.

# Good Examples

Good example:

A framework that moves from technology change to capability gap to execution risk to business consequence.

Why this works:

It directly supports the core brand philosophy.

Good example:

A decision checklist for leaders assessing whether AI training will actually change team behaviour.

Why this works:

It is practical, reusable, and audience relevant.

# Bad Examples

Bad example:

A vague four step framework about growth mindset.

Why this fails:

It does not support the brand position strongly enough.

Bad example:

A copied popular framework renamed with Nayara's colours.

Why this fails:

It risks originality and trust.

# Common Mistakes

• Creating frameworks for the sake of looking strategic.
• Making frameworks too complex.
• Saving ideas before they are reusable.
• Forgetting to explain when to use the framework.
• Presenting borrowed thinking as original.

# Anti Patterns

• Framework clutter.
• Fake originality.
• Abstract models with no decision value.
• One off diagrams stored as frameworks.
• Frameworks that ignore audience reality.
• Systems that sound impressive but do not help.
