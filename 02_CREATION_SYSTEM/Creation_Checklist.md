# Purpose

This document defines the final creation checks that content must pass before it moves into the Review System or is considered ready for publication review.

# How this document connects to the rest of the repository

Creation_Checklist.md is the final gate inside the Creation System. It should be used after drafting and rewriting, but before quality review or approval.

# Core Principles

• Checklist discipline prevents drift.
• A strong draft must still be checked.
• Content must serve the brand position.
• Every claim must be safe.
• Every post should have one clear idea.
• Readability matters as much as insight.

# Decision Framework

Before marking content creation complete, ask:

1. Is the objective clear?
2. Is the reader clear?
3. Is there one idea?
4. Is the hook strong and honest?
5. Is the structure appropriate?
6. Is the voice aligned?
7. Are claims safe?
8. Is the CTA useful or intentionally omitted?

# Detailed Rules

Creation checklist:

1. Foundation consulted.
2. Audience identified.
3. One idea defined.
4. Hook aligned to the post.
5. Structure selected intentionally.
6. Body supports the hook.
7. Voice matches Nayara's brand.
8. Vocabulary is specific and non generic.
9. Claims are labelled or supported.
10. Confidential information removed.
11. CTA is useful or omitted.
12. Mobile readability checked.
13. Content is ready for review.

Rules:

If any item fails, revise before moving forward.

Do not treat a nearly ready draft as approved.

Do not skip knowledge boundary checks.

Edge cases:

If the content is a comment, apply the checklist in compressed form.

If the content is a carousel, check slide logic as well as copy.

If the content is a profile update, check positioning before voice.

# Good Examples

Good example:

A draft is strong but contains two ideas. It is split into two future posts.

Why this works:

It protects clarity and future content supply.

Good example:

A CTA is removed because the final sentence is already strong.

Why this works:

It avoids weakening the ending.

# Bad Examples

Bad example:

Approving a post because the hook is strong, even though the body does not support it.

Why this fails:

It creates reader disappointment and weakens trust.

Bad example:

Keeping a statistic from memory.

Why this fails:

It violates knowledge boundaries.

# Common Mistakes

• Treating checklist work as optional.
• Keeping too many ideas.
• Forgetting mobile readability.
• Adding a CTA by habit.
• Keeping vague language because the idea feels strong.
• Skipping confidentiality checks.

# Anti Patterns

• Approval without checklist.
• Hook body mismatch.
• Unsupported claims.
• Generic language.
• Weak CTA.
• Content that ignores the positioning.
