# Purpose

This document defines the review process every Brand OS document and content output must pass before approval. It protects quality by forcing critique, rewrite, stress testing, and validation before anything is treated as finished.

# How this document connects to the rest of the repository

Review_Loop.md is the first document in the Review System. It connects the repository standards from 00_START_HERE with the creation process in 02_CREATION_SYSTEM.

No folder, document, or public output should be considered approved until this loop has been completed.

# Core Principles

• Review is not optional.
• A strong first draft is still a draft until reviewed.
• Critique must happen before approval.
• Rewriting must resolve real weaknesses, not only polish wording.
• Stress testing should look for drift, hallucination, inconsistency, and weak strategy.
• Approval means the output is safe to use, not merely acceptable.

# Decision Framework

Before approving any output, ask:

1. Does it follow the relevant Brand OS standards?
2. Does it support the correct brand position?
3. Does it contain unsupported claims?
4. Does it sound like Nayara's voice?
5. Does it have one clear purpose?
6. Does it create any confidentiality or accuracy risk?
7. Would it still make sense outside the current conversation?
8. Is it strong enough to enter the repository or be published?

# Detailed Rules

Review loop sequence:

1. Read relevant dependencies.
2. Check metadata if the output is a repository file.
3. Check mandatory structure if the output is a Markdown document.
4. Assess strategic alignment.
5. Critique the substance.
6. Critique the structure.
7. Critique the voice.
8. Check for hallucination, unsupported claims, or confidential details.
9. Rewrite to fix issues.
10. Stress test the revised version.
11. Run the relevant checklist.
12. Approve only if all checks pass.

Rules:

Do not approve a file because it is useful if it breaks the repository standard.

Do not approve content because it sounds good if it lacks strategic alignment.

Do not approve claims that cannot be supported, labelled, or safely softened.

Do not move to the next folder if the current folder has unresolved issues.

Edge cases:

If the user asks for speed, complete the review loop internally in compressed form.

If the output is a short message, use a lighter review but still check voice, accuracy, and purpose.

If the output is repository content, use the full review loop.

If a document fails after revision, do not save it.

# Good Examples

Good example:

A Markdown file has strong content but includes an extra section. It is rejected and rewritten into the mandatory structure.

Why this works:

Repository consistency is protected.

Good example:

A LinkedIn post has a strong hook but contains an unsupported statistic. The statistic is removed or replaced with a labelled observation.

Why this works:

Trust is protected.

Good example:

A document is strategically correct but sounds too corporate. It is rewritten using Voice_Communication_System.md.

Why this works:

Brand identity is protected.

# Bad Examples

Bad example:

Approving a document because the idea is strong even though it misses mandatory sections.

Why this fails:

It creates repository drift.

Bad example:

Approving a post because it is catchy even though it sounds unlike Nayara.

Why this fails:

Attention is treated as more important than brand consistency.

Bad example:

Leaving a remembered statistic in the content because it sounds credible.

Why this fails:

It increases hallucination risk.

# Common Mistakes

• Treating review as proofreading.
• Fixing tone but ignoring strategy.
• Fixing strategy but ignoring structure.
• Approving useful content that violates standards.
• Skipping hallucination checks.
• Confusing polished with approved.
• Moving forward before the folder is truly complete.

# Anti Patterns

• Approval without critique.
• Cosmetic rewrites.
• Silent assumption making.
• Checklist skipping.
• Standards applied differently across folders.
• Saving unfinished work into the repository.
• Treating good enough as repository ready.
